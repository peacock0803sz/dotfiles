# 技術文書向けの追加規範

長文の技術原稿(記事、解説、書籍の章)を校正するときに、本体の`SKILL.md`の規則に加えて適用する。短い文章やメモ、チャット向けの文には使わない。

出典は k16shikano「japanese-tech-writing」(https://gist.github.com/k16shikano/fd287c3133457c4fd8f5601d34aa817d)。技術文書に固有の項目だけを抜き出して再構成した。

## 整形

- 用語の由来や定式化の名称など、本筋から一段外れる補足は、本文に並べず脚注(`[^ラベル]`)に降ろす。
- コードや差分、ログ、設定ファイルの断片はコードブロックで示す。
- コラム見出しは「基礎」「補足」のような種別名だけで済ませず、「同値関係としての分類」「ループ不変条件と帰納法」のように内容を特定する。

## 見出しの付け方

- 見出しは内容を特定できる具体的な句にする。その節が答える問い、または扱う対象を指す。
- 作業の手順だけを述べる見出し(「例に戻す」など)や、情報量のない見出しにしない。
- 見出しを、節の結論を言い切る「セリフ」にしない。見出しの時点で読者がオチを知る状態を避ける。
- 区切り線(罫線やダッシュ類)で「種別──主題」のように二要素を詰め込まない。単一の自然な句にする。
- 疑問形にするか対象を指す名詞句にするかは、本文のトーンに合うほうを選ぶ。

## 章と節をまたぐ整合

- 同じ概念の扱いを全体で一致させる。ある節で「人間が決める」と分類したものを、別の節で「チームで合意する」と書かない。分類や定義、用語の地位を揃える。
- 節の中心となる語は、その節以前に定義と対象範囲を述べてから使う。定義せずに使い始めない。

## 前方参照と例への弁明

- 「後の章で扱う」のような前方参照は、論証が一段落した位置(段落末や節末)に置く。論証の途中に挟んで流れを切らない。
- 前方に逃がした論点は、本当にそこで回収されることを確認する。回収しない伏線を張らない。
- 例への弁明(作為的に見えるという疑念への先回りなど)は、場面の山場の直後に挟まず、次の節の冒頭でまとめて処理する。

## 術語と訳語、人名の表記

- 術語や訳語は、その分野で慣用されている語を選ぶ(プッシュ通知は「配送」でなく「配信」など)。意味の近い漢語を一般語の感覚で充てない。
- 術語の響きを持つ語を、術語でない場面に流用しない。「届くまでの流れ」のように普通の言い方で書く。
- 人物そのものに言及するときは原綴りで書く(Lehman、Bainbridge)。ただし、歴史上の人物や人名を冠した概念を定着した名前で紹介するときは、日本語で通用しているカタカナの通称を使う。
