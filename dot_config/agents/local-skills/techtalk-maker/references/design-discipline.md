# デザイン規律の良い例と悪い例

スライドの見た目は、内容の説得力に直結する。整え方の最低限を例で示す。

## 文字サイズ

16:9のHD解像度(1280x720px)で次のサイズをキープする。

- 本文 24〜28px
- 見出し 30〜36px
- コード 22〜26px (等幅フォント)

これより小さいと後列から読めない。発表会場のスクリーンは想定より遠く、後ろの列の人は文字が点にしか見えない。

slidevなら `class: text-2xl` 以上、Marpなら`<style>section { font-size: 28px; }</style>` で指定する。pptxは「フォントサイズを28pt以上」をテンプレートに固定する。

### 悪い例

```
# 何かの発表
- 細かい話を箇条書きで書く 12pt
- 図表のキャプションも 10pt
- 注釈は 8pt
```

12ptは小さすぎる。後列から読めず、聴衆は内容を理解する前に読むことを諦める。

### 良い例

```
# 何かの発表
- 大事なことだけを大きく 28pt
- 補足は本文と同じか少し小さく 24pt
```

## 色数

色は1〜2色に絞る。

- 文字色 黒またはダークグレー
- 強調色 1色だけ(青か赤か緑のどれか1つ)
- 背景色 白かオフホワイト

これに加えて、コードのシンタックスハイライトは別扱いとしてよい。

3色以上にすると、強調が分散して何が大事か伝わらない。「赤=エラー, 黄=警告, 緑=正常, 青=情報」のように意味づけしても、聴衆は意味を覚えるコストを払えない。

### 悪い例

タイトルが青、見出しが緑、本文が黒、注釈が灰、警告が赤、リンクが紫。6色。聴衆の視線が散る。

### 良い例

すべて黒。強調したい単語1つだけが青。引用も黒のまま、下線で示す。

## 整列

見出しと注釈の上下を揃え、行頭の左右も揃える。

- 見出しの位置を全スライドで同じy座標に置く。
- 箇条書きの左端を全スライドで揃える。
- 注釈(ページ番号, 引用元)を全スライドで同じ位置に置く。

揃っていないと、ページ送りのたびに目がずれて疲れる。

slidevはテーマがこれを担保する。テーマを変えず`default`のまま使うのが最も安全。Marpも同様。pptxは「スライドマスター」を1つ作って全スライドに適用する。

## 1枚1メッセージ

要点だけスライドに残し、詳細はspeaker noteに逃がす。

### 悪い例

```
# 認証フロー

1. ユーザーが認証ボタンを押す
2. クライアントがOAuthエンドポイントへリダイレクト
3. ユーザーがログイン情報を入力
4. OAuthプロバイダーがコールバックURLにリダイレクト
5. クライアントが認可コードを受け取る
6. クライアントがアクセストークンを取得
7. クライアントがトークンを使ってAPIを呼ぶ
8. APIがトークンを検証
9. APIがデータを返す
10. クライアントがUIを更新
```

10行を聴衆が読み終わるまで30秒かかる。その間発表者は何を話せばよいか。

### 良い例

スライド本体

```
# 認証フロー

クライアントとAPIの間に、OAuthプロバイダーが1段挟まる
```

speaker note

```
<!--
具体的な流れは10ステップ。
ユーザーボタン押下 → OAuthへリダイレクト → ログイン → コールバック → 認可コード → トークン交換 → API呼び出し → 検証 → データ → UI更新。
ここは口頭で流す。図はあとのスライドで出す。
-->
```

スライドは「OAuthプロバイダーが1段挟まる」という主張だけ残す。詳細は口頭で流すかspeaker noteを後で読んでもらう。

## フォント

標準的なゴシック体1つに統一する。

- 英文 Inter, Helvetica, system-ui
- 和文 Noto Sans JP, Hiragino Sans, Yu Gothic

装飾フォント(明朝, 手書き風, ポップ体)は本文では使わない。見出しでも避ける。

slidevなら frontmatter で次のように指定する。

```yaml
fonts:
  sans: 'Inter, "Noto Sans JP", sans-serif'
  mono: 'JetBrains Mono'
```

## 画像と図

- 画像は本文と同じ視覚的重さに収める。スライドの半分以上を占める巨大画像は本題から目を逸らす。
- 図は説明する要素だけ残す。AWSアーキテクチャ図の全コンポーネントを並べるより、議論する3つのコンポーネントだけ残す方が伝わる。
- 図のキャプションは本文より小さく(18〜22px相当)、本文と同じ色にする。

## 半角英数字と日本語の間の空白

slidevやmarpはMarkdownを内蔵CSSでレンダリングする。CSS側がフォントの字幅と日本語との余白を制御するため、Markdownソース上で半角英数字の前後に空白を入れると、本番表示では余白が二重になる。

詳細な規則と修正は proofread-ja スキルに委ねる。SKILL.md の「仕上げに proofread-ja を当てる」のステップで全ファイルに適用すれば、日本語と半角英数字の間の余白、記号の使い方、句読点まで一括で整う。techtalk-maker 側で個別の表記ルールを並べない。

簡単な目安だけ示すと、"URL パス"ではなく"URLパス"、"3 秒"ではなく"3秒"、"TypeScript 4.1以降"のように半角どうしの空白は残す。

## 末尾スライド

最後のスライドは主題の再掲かまとめにする。聴衆の最後の印象がここで決まる。

### 悪い例

末尾2枚を次のように作る。

```
スライド N-1  # 参考リンク
- リンクA
- リンクB
- リンクC

スライド N    # Q&A
ご質問をどうぞ
```

参考リンクスライドは公開資料の補助として作りたくなるが、本文中に都度リンクしていれば不要。Q&Aスライドは情報密度ゼロで枠だけ消費する。発表時間も30秒程度減る。

### 良い例

末尾1枚にまとめる。

```
スライド N    # まとめ

主題の再掲文を1行で。

(必要なら参考リンクをここに3〜5行で添える)
```

質疑応答に入るときは口頭で「ご質問あればどうぞ」と促せばよい。想定質問と回答はspeaker noteに書く。
